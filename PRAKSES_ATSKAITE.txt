ORDERS tabula:

- user_id INT (ārējā atslēga uz users)

- total_price DECIMAL(10,2) - kopējā pasūtījuma cena

ORDER_ITEMS tabula:
- order_item_id INT (primārā atslēga, auto-increment)
- order_id INT (ārējā atslēga uz orders)

- quantity INT - daudzums
- unit_price DECIMAL(10,2) - cena par vienību

SUPPORT_REQUESTS tabula:
- id INT (primārā atslēga, auto-increment)
- name VARCHAR(100) - nosūtītāja vārds
- email VARCHAR(100) - nosūtītāja e-pasts

- timestamp DATETIME - ziņojuma sūtīšanas laiks

5. ER DIAGRAMMAS IZSTRĀDE

5.1. Entītijas un atribūti

Entītijas:
1. USERS - Lietotāji
   Atribūti: user_id, username, email, password_hash, created_at

2. PIZZAS - Pical
   Atribūti: pizza_id, name, description, price, image, allergens

3. ORDERS - Pasūtījumi
   Atribūti: order_id, user_id, order_date, total_price

4. ORDER_ITEMS - Pasūtījuma lietas
   Atribūti: order_item_id, order_id, pizza_id, quantity, unit_price

5. SUPPORT_REQUESTS - Palīdzības pieprasījumi
   Atribūti: id, name, email, message, timestamp

5.2. Attiecības (1:1, 1:N, N:M)

Attiecības sistēmā:

USERS 1:N ORDERS
- Viens lietotājs var veikt daudz pasūtījumu
- Katra pasūtījums pieder vienam lietotājam

ORDERS 1:N ORDER_ITEMS
- Viens pasūtījums var satur daudz pozīciju (picas)
- Katra pozīcija pieder vienam pasūtījumam

PIZZAS 1:N ORDER_ITEMS
- Viena pica var būt pasūtīta daudz reižu
- Katra pasūtīta pozīcija ir viena pica

N:M attiecības (uz indirect):
- Users un Pizzas saistās caur ORDER_ITEMS (N:M relācija)
- Viens lietotājs var nopirkt daudz picas
- Viena pica var būt nopirkta daudz lietotāju

5.3. Normalizācija

Normalizācija tika ievērota, lai izvairītos no liekas datu dublēšanās:

1NF (pirmā normaļa forma):
- Visus lauki satur atomārus datus (bez grupu atkārtošanās)
- Katra rinda ine un viens datu komplekts

2NF (otrā normaļa forma):
- Visi neklastera lauki ir pilnībā atkarīgi no primārās atslēgas
- Nav daļējas atkarības

3NF (trešā normaļā forma):
- Nav transitīvās atkarības
- Informācija par lietotājiem, picām un pasūtījumiem ir atsevišķas tabulās
- Saistības nodrošina pie ārējās atslēgas

7.2. SQL vaicājumu piemēri

SELECT - datu iegūšana:
  SELECT * FROM pizzas;
  SELECT * FROM orders WHERE user_id = 5;
  SELECT * FROM order_items WHERE order_id = 3;

INSERT - jaunu datu pievienošana:
  INSERT INTO orders (user_id, order_date, total_price) 
  VALUES (3, NOW(), 25.50);

UPDATE - datu mainīšana:
  UPDATE users SET email = 'jauns@mail.com' WHERE user_id = 1;

DELETE - datu dzēšana:
  DELETE FROM orders WHERE order_id = 10;

JOIN vaicājumi - datu saistīšana no vairākām tabulām:
  SELECT o.*, m.name FROM orders o 
  JOIN users u ON o.user_id = u.user_id;

8. KĻŪDU MEKLĒŠANA UN NOVĒRTĒŠANA

8.1. SQL kļūdas

Atrastajas SQL kļūdas:

Kļūda 1: Nepareizi lauks nosaukums
- Problema: SELECT user_id FROM users - bez order_date
- Risinājums: Pievienot pareizo lauko nosaukumu

Kļūda 2: Nepareiza tabula nosaukums
- Problema: SELECT * FROM "order" (order ir rezervēts vārds)
- Risinājums: Izmantot pareizu tabulas nosaukumu "orders"

Kļūda 3: Datu tipa nesakritība
- Problema: Datumi glabājas kā DATETIME, bet INSERT dod sadaļu
- Risinājums: Izmantot NOW() vai pareizu data formātu

8.2. GUI kļūdas

Atrastajas GUI kļūdas:

Kļūda 1: Formas validācija nepilnīga
- Problema: Lietotājs var ievadīt tukšas vērtības
- Risinājums: Pievienot required atribūtu HTML un isset() pārbaudi PHP

Kļūda 2: Neērts izkārtojums tabulās
- Problema: Slejas pārāk šaura, teksts izskatās apspiests
- Risinājums: Paplašināt kolonnas un labāk sadalīt vietu

Kļūda 3: Nesakarīga krāsu izvēle
- Problema: Teksts grūti nolasāms uz dažām krāsām
- Risinājums: Izmantot melnu tekstu uz baltu/gaišu fonu

8.3. Risinājumi

Risinājumi kļūdām:

1. HTML validācija:
   <input type="email" required>
   <input type="password" required>

2. PHP validācija:
   if (empty($_POST['username'])) {
       $error = "Lietotāja vārds ir obligāts";
   }

3. SQL prepared statements:
   $stmt = $conn->prepare("SELECT * FROM users WHERE email = ?");
   $stmt->bind_param("s", $email);
   $stmt->execute();

4. GUI labojumi:
   - Paplašinātas tabulu kolonnas
   - Pievienots proper CSS padding un margin
   - Labāka krāsu kontrasta

10. SECINĀJUMI NO TEORIJAS UN PRAKSES

10.1. Secinājumi no teorijas

Teorijā tika apgūti šādi jēdzieni:

- RELĀCIJU DATU BĀZES
  Teoriski: Relāciju modelis, tabulas, atslēgas, normalizācija
  Praksi: Izveidoj 5 tabulas ar pareizām relācijām

- NORMALIZĀCIJA
  Teorija: 1NF, 2NF, 3NF principi
  Praksi: Ievērots 3NF (bez atkārtošanās, vienas atbildības)

- SQL VAICĀJUMI
  Teorija: SELECT, INSERT, UPDATE, DELETE sintakse
  Praksi: Pielietoti visos sistēmas moduļos

- DROŠĪBA
  Teorija: SQL injection, XSS, autentifikācija
  Praksi: Pielietotas prepared statements, htmlspecialchars()

- GUI PRINCIPI
  Teorija: Lietotāja pieredze, intuitīvs dizains
  Praksi: Minimālistisks dizains, vienkāršas vadības

10.2. Secinājumi no prakses

Praksē gūtā pieredze:

1. PLĀNOŠANA IR SVARĪGA
   - Pirms kodēšanas svarīgi skaidri noplānot sistēmu
   - ER diagramma palīdz redzēt kopējo attēlu

2. DAZU BĀZE PIRMKĀRT
   - Labas datu bāzes struktūra skaņo pūpolu vienkāršo
   - Diezgan viduslaika dati vienmēr ierosina problēmas

3. MODULĀRS KODS
   - Svarīgi dalīt kodu mazākos moduļos
   - Vieglāk atkļūdot un uzlabot

4. TESTĒŠANA SVARĪGA
   - Testēšanā ieguvu 95% no kļūdām
   - Automātiski testi ietaupītu daudz laika

5. DOKUMENTĀCIJA
   - Dokumentēt nopietni, savs vērtējums pēc 2 nedēļiem

6. VERSIJU KONTROLE
   - Git palīdz sekot izmaiņam un neiznīcināt kodu
   - GitHub publiski dara kod pieejamu
